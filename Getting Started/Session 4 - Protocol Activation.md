# Session 4 - Protocol Activation

Purpose: activate the core protocols and skills that let the Sovereign Ecosystem actually operate instead of remaining a well-organized but mostly passive structure.

By now you have:
- a working local workspace
- a first adapter bridge
- governance you can trust without needing to memorize
- codices that give the system more context and specificity

This session is where the operating loops start to come online.

This is also where a deeper kind of trust starts to become possible.

Before this point, the system may feel thoughtful, promising, even exciting.
After this point, it can start feeling dependable.

You do not need to read every protocol in the ecosystem today.
That would be a fast way to turn a useful system into a bureaucracy cosplay event.

You do want to activate the small subset that makes the Foundation usable.

## Session Goal

By the end of this session you should have:
- understood the difference between protocols and skills
- activated the core Foundation protocols and their related skills
- brought the intake lane online in a lightweight way
- completed the Foundation naming checkpoint if names are still unresolved

## Principle

Protocols are the operating logic.
Skills are the reusable ways your AI interface carries that logic into action.

Protocols explain how the system works.
Skills help the system do the thing repeatedly without you rebuilding the process every time.

The real value here is not technical neatness.
It is that once these sequences are operational, you stop having to reinvent the wheel every time you want the system to do something.

You get to say something, approve something, or start something moving and trust that the sequence will hold.

That trust will not appear by magic on day one.
There will be trial and error.
There will be refinements.
Sometimes the system will be smart and elegant.
Sometimes it will put the metaphorical soap in the refrigerator.

That is not failure.
That is how the collaboration gets better.

The more you use the protocols and skills, the more you refine them.
The more you refine them, the more the system starts feeling like it knows how to dance with you.

This session is not about installing every possible behavior.
It is about turning on the core loops that make the Foundation feel alive and usable.

## Step 2 - Understand Protocols Versus Skills

Open:
- `Council Chamber/Skills/Skills Index.md`

Use this simple distinction:

- a protocol says how a repeatable pattern should work
- a skill is a usable execution package for your AI interface

If a protocol is the choreography, a skill is the rehearsal-ready routine.

Another way to say it:
- the protocol is the trusted sequence
- the skill is the way that sequence gets carried out in practice with your AI interface

You do not need to obsess over the terminology.
Just understand that the ecosystem uses both:
- governance and protocols for structural clarity
- skills for repeatable AI-assisted execution

## Step 4 - Bring The Intake Lane Online

The structural notes treat `Inbox` as an active working lane.
The capture protocol also expects a lightweight quick-capture surface.

Both `Inbox/` and `To-Do Dock.md` are pre-shipped in the Foundation.
Confirm they exist and orient to what they do:
- `Inbox/` is for captured material that may need thought, routing, or conversion
- `To-Do Dock.md` is for quick action prompts you do not want in your head

Useful reference points:
- `Council Chamber/Templates/Object Templates/InboxItem.md`
- `Council Chamber/Protocols/Inbox/Capture Classify Route Protocol.md`

This is where the system can start freeing real mental bandwidth.

Something comes in.
You do not have to hold it all in working memory.
You do not have to remember where it goes every time.
You capture it.
The system helps classify it.
Then it gets routed where it actually belongs.

Think of it like ordering a few things for the house.
You do not really want every package to stay sitting at the front door forever.
You want the vitamins to go where the vitamins go.
You want the soap to go where the soap goes.
You want the thing to end up in the place where it is actually useful.

That is what the intake flow is trying to create for information.

At first, do not over-automate this.
Pattern recognition is strong, but early overconfidence is how soap ends up in the refrigerator.

Start with light trust.
Give feedback.
Let the routing get more accurate through use.

If you mostly use one or the other early on, that is fine.
The goal is not perfect taxonomy.
The goal is a capture habit that the system can support.

Now they are starting to attach to an actual operating environment.

You do not need the most mythic or permanent name imaginable.
You do want names that feel usable enough to carry forward.

**To replace all tokens across the vault, run the one-time replacement script:**

First confirm Node.js is available:
```
node --version
```

Then run:
```
node scripts/replace-tokens.mjs
```

This script will:
- ask for your chosen names
- preview every file that will change before writing anything
- replace all token occurrences and log the results

After the script completes, move it to `.archive/` or delete it.
It is a one-time tool. Once run, it has served its purpose.

If you already renamed the AI interface in Session 1, that token is already handled.
Verify your files look correct — you do not need to run the script again for that field.

Remember:
- naming substitution is enough for now
- deeper behavior customization can happen later
