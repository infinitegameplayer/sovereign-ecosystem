# Flywheel Integration Standard

Purpose: Define how all Sovereign Ecosystem flywheels align with the Flywheel Codex of the Sovereign Creator and how governance integrates them consistently.

## Scope
- Applies to all flywheel protocols, loops and process cycles labeled "flywheel."
- Applies across all domains and Active Chambers when a flywheel governs transitions, archival movement or compounding feedback cycles.

## Canonical Codex
- The primary authority is the Flywheel Codex of the Sovereign Creator.
- All flywheel protocols must reference the codex as their governing doctrine.
- The three sovereign laws are mandatory for all flywheels:
  - Smooth Transition
  - Easier Rotation
  - Increased Output

## Naming Convention
- Flywheel protocol naming format: `X Flywheel Protocol vY.Y`.
- Sovereign Ecosystem-wide canonical flywheel names use the term `Vault` for archival-centric flywheels (e.g., `Universal Vault Flywheel Protocol`).
- Additional flywheels should use concise, single-noun or noun-phrase identifiers (e.g., `North Star Flywheel Protocol`).

## Integration Requirements
1. Each flywheel protocol must include:
   - Purpose
   - Scope
   - Trigger
   - Stages or Loop
   - Approval gate
   - Links to the Flywheel Codex
2. Any protocol that invokes a flywheel must reference the correct flywheel protocol by name.
3. Flywheel protocols may propose deltas to governance artifacts; approvals are required before mutation.
4. Flywheel protocols may not override constitutional rules, consent gates or raw preservation.
5. Flywheel protocols must define how they connect to at least one adjacent flywheel (inputs, outputs or handoff).
6. Cross-flywheel connections must be explicit and named to reduce drift.

## Flywheel Connectivity Map (Current)
- North Star Flywheel -> Quests/Experiments -> Artifacts -> Universal Vault Flywheel
- Quests/Experiments -> Conduction Flywheel -> Universal Vault Flywheel
- Universal Vault Flywheel -> Taste Delta + Protocol Delta signals -> North Star Recalibration and Governance refinement
- Security Flywheel -> Meta-Governance Audit -> Governance refinement + risk hardening proposals
- Embodied Contribution Flywheel (governance architecture) -> body-led aliveness -> `Sovereign Ecosystem Command` (first practical Intelligence report) -> Quests/Collaborations -> Expression/Speaking -> Business / Contribution Lanes -> Wealth byproduct -> periodic Archetypal + Areas of Life review
- Non-protocol flywheels (e.g., Embodiment, Synergy, Creative Spark) should reference a protocol flywheel for operational handoff when formalized.

## Stewardship Threads (Optional)
- Maintain lightweight links from stewardship guidance to relevant Quests/Experiments when they are created.
- These links are directional (guidance -> practice) and do not impose governance constraints.

## Governance Alignment
- Flywheel protocols are governance artifacts and must follow:
  - Plan -> Consent -> Execute
  - Contrast Layer requirements for foundational governance artifacts
  - AI Interface Change Log Protocol for any approved flywheel changes

## External Collaboration Alignment
- When flywheel doctrine is applied outside the Sovereign Ecosystem, exported artifacts or collaboration containers must include:
  - A brief flywheel alignment note
  - The applicable flywheel protocol name(s)
  - A statement that Sovereign Ecosystem governance rules remain sovereign within the Sovereign Ecosystem

## Adoption Sequence
1. Codex anchored in Council Chamber/Codices.
2. Flywheel protocols linked to the codex.
3. Referencing protocols updated to align naming and link targets.
4. Domain or object pilots executed where required.
