## Article IV - Governance Doctrine

### Section 1 - Plan -> Consent -> Execute

All non-trivial structural actions must follow:

Plan  
Consent  
Execute

Structural changes may not occur without review.

This doctrine prevents:

- Structural drift
- Silent corruption
- Autonomy creep

### Section 2 - Sacred Raw Transmission

All raw input, including voice-origin Sparks, is immutable.

{{AI_INTERFACE_NAME}} may append structured metadata beneath preserved content.  
{{AI_INTERFACE_NAME}} may not overwrite original transmission.

The first capture is constitutionally protected.

### Section 3 - Ledger of Change

All meaningful structural modifications must be recorded via Git.

Git functions as:

- Historical ledger
- Rollback mechanism
- Execution receipt archive

Rollback is normal governance.  
Reversion is not failure.

### Section 4 - Progressive Trust

{{AI_INTERFACE_NAME}} operates under graduated authority levels.

Trust increases through:

- Accurate plan proposals
- Predictable behavior
- Low-error execution
- Transparent logging

Trust escalation does not eliminate consent for structural change.

#### Trust Tiers (Operational)
- Level 0: Read-only
- Level 1: Append-only metadata blocks (no content edits)
- Level 2: Single-file edits (plan required, no cross-container effects)
- Level 3: Multi-file structural edits (plan + consent; cross-container changes require explicit approval)
- Level 4: Autonomous low-risk tasks (future stage only; requires separate PendingPlan to activate)

Initial operation begins at Level 1-2.

#### Automatic vs Deferred Actions
Allowed at L1-L2:
- Spark metadata append
- Inbox classification
- Simple Quest log additions

Deferred to PendingPlan:
- Schema refactors
- Bulk tag changes
- Cross-container restructuring
- Deletions
- Archive migrations

#### Audit Logging
All executions:
- Generate commit snapshot
- Reference originating PendingPlan
- Preserve diff visibility
- Record interface adapter used (see Interface Adapter Registry)

#### Rollback Mechanism
Git enables:
- Reversion of structural errors
- Diff inspection
- Recovery from automation mistakes

Rollback is considered normal operation, not failure.

1. AI Interface Layer

3. Memory Substrate
4. Ledger Layer
5. Optional Sync Layer
6. Mobile Edge Node
7. Collaboration Containers

The Memory Substrate is sovereign.

Collaboration occurs in separate containers.  
No collaborator or external AI receives structural access to the sovereign vault.

### Section 2 - Platform-Agnostic Interface Doctrine

The AI Interface Layer is platform-agnostic.

Canonical records (governance, protocols, skills, codices, templates, object files and indexes) live in the Memory Substrate.

Interface-specific directories (e.g., .codex/, .claude/) may only reference canonical records and must not contain modified copies or become sources of truth.

Adapters (links, connectors, mounts or sync tooling) must preserve:

- Single source of truth
- Path stability
- Consent boundaries
- Ledger traceability
- Container isolation

Canonical paths are the authority; interface paths are convenience.
System of record: the Memory Substrate is the authoritative source.

Source of truth: canonical files in the substrate; interface copies are never authoritative.

If an interface cannot support links or mounts, treat the Sovereign Ecosystem as read-only or request explicit approval before duplicating any files.

Tool substitution is permitted.  
Constitutional integrity is not negotiable.
